# tech-article スキルの設計ドキュメント

このスキルは、バーバラ・ミント『考える技術・書く技術』の原則に基づく技術記事執筆支援を目的とする。設計の主要決定を集約するドキュメント。実装の途中でコンテキストをクリアする場合、このドキュメントと memory を読み込めば作業を再開できる。

## スキルの全体像

ユーザーが技術記事を書く際、企画から公開準備までを対話で支援する。単一のスキル本体（オーケストレーター）がフェーズ制御を行い、入出力が明確な検査系タスクはサブエージェントに分離する。

ユーザーは `/tech-article` で明示起動するか、「記事を書きたい」などの発話でスキル description のキーワードマッチにより自動起動する。

### スキルの段階（フェーズ）

オーケストレーター内部で扱うフェーズは以下。対話型の 3 フェーズと、サブエージェントで実行する 2 種類の検査タスク、そして公開準備からなる。

- **企画**: 対話型。R1/R2 ギャップ + 3 質問で記事の問題定義を固める（`references/planning-framework.md`）
- **構造**: 対話型。トップダウン方式の 8 ステップでピラミッドを組み立てる（`references/structure-topdown-8steps.md`）
- **導入**: 対話型。SCQA の標準形 + 変形パターンで導入部を設計する（`references/intro-scqa.md`）
- **検証**: サブエージェント（`.claude/agents/tech-article-validator.md`）。ピラミッドの論理・MECE・白紙の主張・イシュー曖昧さなどをチェック
- **推敲**: サブエージェント（`.claude/agents/tech-article-polisher.md`）。見出しパラレル化・3 行ルール・キーワード抽出テストなどで文章レベルの改善提案
- **公開準備**: 対話型。HTML コメント除去と Zenn frontmatter 生成（`references/publish-zenn.md`）

対話 3 フェーズの基盤知識として `references/pyramid-core.md`（ピラミッドの中心命題・認知的根拠・3 関係性・再帰性）と `references/logic-rules.md`（演繹/帰納・3 順序・要約品質・「単なる情報」判定）が存在し、検証サブエージェントも `logic-rules.md` を参照する。

書籍からの具体例（チューリッヒの髭の男たち、豚はペットとして等）は `references/book-examples.md` に集約し、他の references からは参照のみ。

## ファイル構成

```
.claude/skills/tech-article/
  DESIGN.md                              # このファイル
  SKILL.md                               # オーケストレーター本体
  references/
    pyramid-core.md                      # ピラミッドの中心命題・認知根拠・3 関係性・再帰性
    planning-framework.md                # R1/R2・4 要素フレーム・3 質問
    structure-topdown-8steps.md          # トップダウン 8 ステップ + ボトムアップ 3 ステップ
    intro-scqa.md                        # SCQA 標準形 + 4 変形パターン + 失敗警告
    logic-rules.md                       # 演繹/帰納・3 順序・MECE・要約品質
    book-examples.md                     # 書籍内の代表例の集約
    publish-zenn.md                      # Zenn frontmatter + HTML コメント除去
.claude/agents/
  tech-article-validator.md              # 論理検証サブエージェント
  tech-article-polisher.md               # 推敲サブエージェント
```

合計 10 ファイル + DESIGN.md。

## 状態管理

フェーズ往復を許容する設計のため、執筆中の状態を記事ファイルに持たせる。場所は記事本文の末尾に埋める HTML コメント。成果物（現在の状態）と履歴（遷移と判断の記録）の二層で構成する。

成果物は各フェーズの出力を構造化して格納する。たとえば企画フェーズなら R1・R2・スタートポイント・懸念される出来事、構造フェーズなら主題・主ポイント・キーラインのリストといった具合。導入フェーズの SCQA 型（標準型/直接型/懸念型/積極型）や、検証・推敲の最終実行日時と指摘事項もここに入る。

履歴は日時 + フェーズ + 短い判断理由の 1 行エントリを追記していく。書き手がどのフェーズに遷移し、そこで何を判断したかを時系列で追える。フェーズ内の細かな対話や個別の文言調整は履歴には残さず、成果物の差分で見えるようにする。

公開準備フェーズで記事を仕上げる際に、このコメントブロック全体を除去する。

### フェーズ往復時の扱い

ユーザーがすでに完了したフェーズに戻って成果物を変更した場合、下流のフェーズ成果物に staleness マーキングを付与する（強制無効化はしない）。マーキングの対象は、変更したフィールドの意味的重要度でフィールド単位で事前に分類する。

重要（変更で下流を stale マーキングする）に分類するのは、企画の R1・R2・スタートポイント・懸念される出来事、構造の主題・主ポイント、キーラインの追加/削除/入れ替え、導入の SCQA 型変更、疑問/答えの本質的変更。それ以外（導入ドラフトの言い回し、細かい推敲）は下流に影響しないとみなす。

## オーケストレーターの振る舞い

### 起動時

ターゲットとなる記事ファイルを識別し、その HTML コメントから状態を読み取る。記事ファイルが特定できない場合はユーザーに確認する。進捗がある場合は現在状態を提示したうえで、「次に進む / 前に戻る / 別フェーズへ / 検査を実行する」を細かめに選ばせる。進捗がない場合も同様に、企画から始めるか特定フェーズに直接入るかをユーザーに選ばせる。

### フェーズ遷移

各フェーズ完了時に、オーケストレーターは次に何をするかをユーザーに提案する。線形に進むだけでなく、前に戻る・別フェーズに飛ぶ選択肢を常に提示する。構造フェーズ完了時、導入フェーズ完了時、本文執筆完了時、ユーザーが「完成」と宣言した時のそれぞれで、検証・推敲サブエージェントの実行を提案する（押し付けはしない）。

### サブエージェントの呼び出し

検証・推敲は Task ツール経由で呼び出す。サブエージェント定義ファイルに各ルールが入っているため、オーケストレーターは記事ファイルのパスと状態を渡してタスク実行を依頼する。戻り値は構造化された指摘リスト + サマリ（深刻度付き）の形式。推敲は指摘だけでなく具体的な修正案を提示するが、自動適用はせずユーザー確認を挟む。

## 書き方のルール

### ルールに理由を併記する

reference ドキュメントや判定ルールは、ルール本体だけでなく「なぜそうするのか」を必ず併記する。Claude がエッジケースで柔軟に判断できるため（`memory/feedback_rules_with_reasoning.md`）。

### 散文で書く

分類テーブル（定義 / 役割 / 条件 / 判定 のような見出し）ではなく、流れのある散文で原則を展開する。LLM は文脈の連続性で理解したほうがエッジケースで柔軟に機能するため（`memory/feedback_prose_over_templates.md`）。

箇条書きは並列列挙が本質的な情報（4 つの変形パターンなど）のときだけ使う。強調したい原則は太字や明示的な命題の形で書く。

### 章番号は書かない

書籍の章番号（Ch1, Ch2 のような表記）は reference 本体に書かない。書籍からの直接引用・代表例については `book-examples.md` を参照させる。理由は、章番号を覚えていない読者（将来のユーザー自身を含む）にはノイズで、Claude の意思決定にも関与しないから。

### memo/ を参照しない

抽出フェーズの作業用の `memo/` ディレクトリは、スキル内の reference からは参照しない。抽出物は書籍由来の知識であり、スキルが動作時に参照すべき唯一の知識源は `references/` 配下。

レイヤ順に実装する（references/ を先に書き、続いてサブエージェント、最後に SKILL.md で統合）。1 ファイルずつ書き、ユーザー確認を挟みながら次へ進む。

## 主要な用語と略称

- **R1 / R2**: 望ましくない結果 / 望ましい結果。問題定義フレームの中核
- **主ポイント / キーライン / サポートするポイント**: ピラミッドの階層
- **縦の関係 / 横の関係 / 導入部のストーリー展開（SCQA）**: ピラミッドが成立する 3 つの関係性
- **標準型 / 直接型 / 懸念型 / 積極型**: SCQA の 4 変形パターン
- **白紙の主張**: 「N つの〇〇」だけで中身のない無内容な要約
- **MECE（メシー）**: Mutually Exclusive, Collectively Exhaustive
- **ミニ・プロセス**: 手順の中に内包される別の手順群（ピラミッド再帰性の一形態）
