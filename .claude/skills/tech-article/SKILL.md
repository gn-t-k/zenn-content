# tech-article オーケストレーター

技術記事の執筆を企画から公開準備まで支援する。各フェーズの詳細ルールは `references/` 配下のファイルに分離されており、オーケストレーターはフェーズ制御と状態管理を担う。

## フェーズの全体像

| # | フェーズ | 形式 | 参照リファレンス |
|---|---|---|---|
| 1 | 企画 | 対話 | `references/planning-framework.md` |
| 2 | 構造 | 対話 | `references/structure-topdown-8steps.md` |
| 3 | 導入 | 対話 | `references/intro-scqa.md` |
| 4 | 検証 | サブエージェント | `tech-article-validator` |
| 5 | 推敲 | サブエージェント | `tech-article-polisher` |
| 6 | 公開準備 | 対話 | `references/publish-zenn.md` |

基盤となる概念と論理ルールは `references/pyramid-core.md` と `references/logic-rules.md` に集約されている。書籍からの代表例は `references/book-examples.md` に、Zenn 固有の運用は `references/publish-zenn.md` にある。

## 起動時の手順

1. **ターゲット記事の特定**: 対象となる記事ファイル（`articles/<id>.md`）を特定する。ユーザーが記事ファイルを明示していなければ、既存の下書きから選ぶか新規作成するかを問う。新規作成は Zenn 慣例として `npx zenn new:article` で ID 付きファイルを生成する
2. **状態の読み取り**: 記事末尾の HTML コメントブロックから現在の成果物と履歴を読む。無ければ「新規スタート」として扱う
3. **フェーズの提示**: 現在状態を踏まえて、「次に進む / 前に戻る / 別フェーズへ / 検査を実行する」の 4 択を細かめに提示する。ユーザーが選んだフェーズに入る

## 状態管理

記事ファイルの末尾に HTML コメントとして状態を保持する。成果物（構造化された現在状態）と履歴（1 行の遷移ログ）の二層構造。

```markdown
<!-- tech-article-state
## 成果物
### 企画
- スタートポイント: <記述>
- 懸念される出来事: <記述>
- R1: <記述>
- R2: <記述>
- 読者がすでに試したこと: <記述>

### 構造
- 主題: <記述>
- 主ポイント: <記述>
- キーライン:
  - <キーライン 1>
  - <キーライン 2>

### 導入
- SCQA 型: <標準型 / 直接型 / 懸念型 / 積極型>
- S: <記述>
- C: <記述>
- Q: <記述>
- A: <記述>

### 検証
- 最終実行日時: <日時>
- 未対応の指摘: <件数と概要>

### 推敲
- 最終実行日時: <日時>
- 未対応の指摘: <件数と概要>

## 履歴
- <日時> <フェーズ> <判断理由>
- <日時> <フェーズ> <判断理由>
-->
```

各フェーズの完了時に成果物を更新し、履歴に 1 行追記する。細かな対話内容は残さず、成果物の差分で見えるようにする。公開準備フェーズで `published: true` に切り替える際に、このブロック全体を削除する。

### フェーズ往復時の staleness マーキング

ユーザーがすでに完了したフェーズに戻って成果物を変更した場合、下流のフェーズ成果物に `(stale: 上流が変更されたため再検討推奨)` のマーキングを付与する（強制無効化はしない）。マーキング対象となる重要フィールドは、企画の R1・R2・スタートポイント・懸念される出来事、構造の主題・主ポイント、キーラインの追加/削除/入れ替え、導入の SCQA 型変更、疑問/答えの本質的変更。それ以外（ドラフトの言い回し、細かい推敲）は下流に影響しない。

## 各フェーズの動作

対話型の 4 フェーズ（企画・構造・導入・公開準備）は、それぞれ対応する reference に従って進める。各 reference には対話フローの指針も含まれている。

- **企画**: `references/planning-framework.md`
- **構造**: `references/structure-topdown-8steps.md`
- **導入**: `references/intro-scqa.md`
- **公開準備**: `references/publish-zenn.md`

サブエージェント型の 2 フェーズは Task ツール経由で起動する。

- **検証**: `tech-article-validator` に記事ファイルのパスと状態（成果物）を渡し、構造化された指摘リスト + 深刻度付きサマリを受け取る。戻り値をユーザーに提示し、対応方針（修正する / そのまま進む / 後回し）を問う。**自動修正はしない**
- **推敲**: `tech-article-polisher` に記事ファイルのパスを渡し、指摘 + 具体的な修正案（`before`/`after`）を受け取る。修正案ごとにユーザー承認を得てから適用する。**自動適用はしない**

## フェーズ遷移の原則

- 各フェーズ完了時に、線形に進むだけでなく **前に戻る / 別フェーズに飛ぶ / 検査を実行する** 選択肢を常に提示する
- 構造フェーズ完了時、導入フェーズ完了時、本文執筆完了時、ユーザーが「完成」と宣言した時、のそれぞれで検証・推敲サブエージェントの実行を **提案** する（押し付けない）
- ユーザーが特定フェーズに直接入りたい場合は、前提フェーズの成果物が不足していても入ることを許可する。ただし情報不足で進めなくなったら前フェーズに戻る選択肢を示す
