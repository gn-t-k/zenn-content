# ピラミッド構造のコア原則

ピラミッド構造の中心命題と内部構造を定義する。他の references ファイルおよび SKILL.md の判定ロジックが依拠する基盤。

## 中心命題

> 最もわかりやすい文書とは、情報を**矛盾なくトップダウン型で表現したもの**である。伝えるべき考えは事前に**ピラミッド型に構成**する。

技術記事を書くときは常にこの前提に立つ。結論を先に出し、その下位に根拠・詳細を配置するトップダウン構造が、読み手の理解を最も促す。

### なぜピラミッドなのか

この主張の根拠は、**人間が情報を処理する際の「頭の中の要求」** にある。技術記事の読者にも同じ認知メカニズムが働くので、ピラミッド原則は技術記事にもそのまま適用できる。

人間の頭には、関連性を持つ物事を自動的に同じグループとして認識し、情報をピラミッド状に並び替えようとする性質がある。だから書き手が明示的にグループ化しなくても、読み手は勝手にグループ化する。問題は、**書き手の意図しないグループ化になる可能性が常にある** ということ。だからこそ、書き手側が先回りして明示的なグループ化を示す必要がある。

さらに、短時間で同時に記憶しておける事柄は最大 7 つ程度（ミラーのマジックナンバー 7）という制約がある。情報が多くなれば、読み手は論理的な分類に基づくグループ化に頼らざるを得ない。**同階層の項目数には実質的な上限がある** ということで、技術記事の見出し数やリスト項目数の設計ガイドラインとして効く（具体的な上限は演繹 4 / 帰納 5、詳細は `logic-rules.md`）。

そして何より、読み手は考えをトップダウンに記憶しようとする。全体を要約する考えを先に提示してから個々を説明する順序が、頭のメカニズムに最も合致する。逆に **結論を後回しにすると、読み手は記憶保持のために認知コストを払い続ける** ことになる。技術記事で「最後まで読まないと結論が分からない」構成は、この原則に真っ向から反する。

### ピラミッド構造でない場合に起きること

ピラミッド構造を持たない文書は、**認知コストを読み手に転嫁するだけでなく、誤解される**。読み手は自分で情報の関連性を探し、文中に表現されていない部分を「エネルギーをふりしぼって」穴埋めしようとする。最悪、メッセージがまったく伝わらないまま「時間の無駄」に終わる。この失敗の典型として「チューリッヒの髭の男たち」の例がある（詳細は `book-examples.md`）。

技術記事に置き換えると、「はじめに」や「背景」で関係性を明示せずに情報を列挙すると、読者は勝手に文脈を推測する。書き手が意図した読解を得るには、ピラミッドの頂点（主題・結論）を先に示す必要がある。

## ピラミッドが成立する 3 つの関係性

ピラミッドが「ピラミッド型」と呼べるのは、次の 3 つの関係性がすべて揃っているときだけ。どれか 1 つでも欠ければ、それはピラミッドではない。

**1 つ目は縦の関係**。上位のメッセージが読み手に「なぜ?」「どのようにして?」という疑問を呼び起こし、そのすぐ下のグループがその疑問に答える。これが上から下まで連鎖する。Q&A（質疑応答）の対話形式がピラミッド全体を貫いている、と言い換えてもいい。この関係性がないと、下位の項目は上位と無関係に列挙されているだけになり、読み手は「なぜこの情報が出てきたのか」を自分で推測しなければならなくなる。この連鎖をどこまで続けるかは、**「論理的に考えて読み手がこれ以上の質問をしないところまで」** が判断基準。技術記事では、想定読者が「まだ分からない」と感じる限り下位を展開し、「もう聞きたいことがない」と感じたら止める。

**2 つ目は横の関係**。同じ階層に並ぶメッセージ同士が、**帰納的論理か演繹的論理か** のいずれかで論理的に結びついていなければならない。並列に並んでいるように見えても、実は「単なる情報の列挙」になっているケースは多く、これはピラミッドとして成立しない。第 2 ポイントが第 1 ポイントの主部または述部にコメントしているなら演繹的、そうでなく同じ「種類を表わす名詞」で括れるなら帰納的、というのが見分け方の入口になる（詳細は `logic-rules.md`）。

**3 つ目は導入部の SCQA**。これは厳密にはピラミッド内部の関係ではなく、**ピラミッドへの入り口** にあたる。**状況 (S) → 複雑化 (C) → 疑問 (Q) → 答え (A)** の順で読み手を物語的に導き、ピラミッドの頂点（主ポイント = 答え）に接続する役割を担う。導入部がないと、書き手は読み手のどんな疑問に答えようとしているのかが自分でも曖昧なまま本論に入ってしまい、ピラミッド全体が「何のために存在するか」を失う（詳細は `intro-scqa.md`）。

### 3 つの関係性は連鎖的に機能する

これら 3 つは独立ではない。実際には **導入 SCQA → 縦の関係 → 横の関係** の順で連鎖的に成立していく。

まず導入部で「この文書が答える疑問」と「その答え（= ピラミッドの頂点）」を確定させる。ここが決まらない限り、下に何を展開すべきかも定まらない。頂点が決まると、その答えを主張するために読み手が抱くだろう「新しい疑問」（「なぜ?」「どのようにして?」）が見えてきて、それに答えるための下位グループが縦の関係として引き出される。そして引き出された下位メッセージ群が論理的に妥当かどうか（帰納的に正しくグループ化されているか、演繹的に筋が通っているか）を横の関係として検証する。

この順序に従うと、技術記事の構築順序は自然に **導入 SCQA → 主ポイント（答え）→ キーライン（縦）→ 各セクション内の論理（横）** になる。逆に、いきなりセクション（キーライン）から考え始めると、そもそもその記事が何のために書かれているのかが決まっていないので、セクション間の並びや粒度の判断基準を失う。

## ピラミッドの階層と再帰性

ピラミッドは、頂点から下に向かって複数の階層を持つ。技術記事の文脈で各階層を整理すると以下のようになる。

最上位に **主ポイント** が 1 つある。これは導入部の「答え」と完全一致する、文書全体の中心メッセージ。その下に **キーライン** と呼ばれる主要論点が並ぶ。キーラインは主ポイントに対して読み手が抱く「新しい疑問」（典型的には「なぜ?」「どのようにして?」）への答えにあたる。キーラインのさらに下に **サポートするポイント** が並び、各キーラインを裏付ける事実・説明・根拠を提供する。

### 重要な性質: ピラミッドは再帰的（フラクタル的）

ピラミッド構造は固定の 3 階層ではない。**各キーラインがそれ自体、下位のサブピラミッドの頂点になりうる**。キーラインが持つ答えが複雑で、それ自体に「なぜ?」「どのようにして?」という新たな疑問を呼び起こすなら、そのキーラインを頂点とする小さなピラミッドを作り、さらに下位のサポートを展開する。そのサブピラミッドも、内部で縦の Q&A と横の論理的結びつきが成立していなければならない。

この再帰は「論理的に考えて読み手がこれ以上の質問をしないところまで」続く。技術記事では、想定読者の知識レベルに応じて深さが決まる。初心者向けなら深く、熟練者向けなら浅く。同じ主題でも想定読者が違えば到達すべき深さが変わる。

「ミニ・プロセス」という概念もこの再帰性の一形態で、手順の中に別の手順群が内包されているケース。階層化せずに平坦に展開すると、ポイント数が過剰になり読み手の認知負荷が跳ね上がる（マジックナンバー 7 の制約に衝突する）ので、自然な階層としてサブピラミッドを切り出すのが正しい対処になる。

後続の references や検証ルールで参照される書籍固有の用語。

| 用語 | 意味 |
|---|---|
| ピラミッド型 / トップダウン型 | 結論を先に、下位に根拠を展開する構造 |
| 頭の中の要求 | 人間が情報処理で使う認知メカニズムの総称 |
| 共通の宿命 | 関連する物事を同じグループにする人間の認知的傾向 |
| マジックナンバー 7 | 短時間記憶の上限（ミラーの理論） |
| グループ化 | 関連する情報を 1 つの上位メッセージの下に束ねる作業 |
| Q&A（質疑応答）の対話形式 | 上位と下位の縦の関係 |
| 縦の関係 / 横の関係 / 導入部のストーリー展開 | ピラミッドが成立する 3 つの関係性 |
| 主ポイント | ピラミッドの頂点。文書の主メッセージ |
| キーライン | 主ポイントへの「新しい疑問」に直接答えるメッセージの並び |
| サポートするポイント | キーラインを裏付ける下位の説明 |
| 同じ場所に立つ | 書き手と読み手が共通認識を持つこと |
| ミニ・プロセス | 手順の中に内包される別の手順群（ピラミッド再帰性の一形態） |
