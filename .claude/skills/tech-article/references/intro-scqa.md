# 導入部の SCQA 設計

ピラミッドの頂点と本論をつなぐ導入部を、**状況 (S) → 複雑化 (C) → 疑問 (Q) → 答え (A)** のストーリー形式で構成するためのリファレンス。標準形の各要素の設計原則、4 つの変形パターン、失敗警告を扱う。SCQA がピラミッド全体と果たす役割は `pyramid-core.md`、SCQA の素材となる問題定義は `planning-framework.md` を参照。

## 導入部が果たす役割

導入部は **ピラミッドへの入り口** であり、本論を読み始める前に書き手と読み手が「同じ土俵に立つ」ための助走。ここで「この文書が答える疑問」と「その答え（= 主ポイント）」を確定させないと、下に何を展開すべきかが定まらない。

導入部の本質的な性格は、**「知識を与えるためのものではなく、思い起こさせるためのもの」**。だから導入部には読み手がすでに知っている or 知っていると考えてさしつかえない内容だけが入り、図表や長大な説明は入らない。**図や表を使っているなら、明白なことをくどくど書きすぎている証**、というのが書籍の判定。技術記事で「はじめに」に技術の歴史を延々と書いたり、前提知識を詳説する章を置いたりするのはこの原則に反する。

## 各要素の設計原則（標準形 S→C→Q→A）

### 状況 (Situation)

状況に書くのは、**主題に関して読み手がすでに知っている、または知っていると考えてさしつかえないこと**。その記述だけで意味が伝わる **自己完結的** なものであり、読み手が自動的に合意する **議論の余地がないポイント** から始める。

なぜ状況から始めるのかには心理学的な根拠がある。本文で意見が分かれるかもしれない内容を伝える前に、まず合意するとわかっているものから伝えることで、読み手を柔軟な態度にし、よどみなく読み始めさせる。これによって読み手を特定の時間と場所に誘い込み、次の情報展開への下地を作る。技術記事なら、読者がすでに使っている技術スタックや既存の実装、広く認知されている前提（企画フェーズのスタートポイントに対応）がここに入る。

### 複雑化 (Complication)

複雑化は **単なる「問題」ではなく、状況の中で起こる「緊張を発生させ、疑問の引き金となる」事象**。状況で描いた安定した世界を揺さぶり、読み手の頭に「どうすれば?」という疑問を生じさせる出来事。

状況と複雑化を繋ぐ判定基準が **So What? テスト**。状況を読んだ読み手が「なるほど、確かに。それで何なの？」と反応する、そのきっかけを捉えて「次に何が起こったか」へ展開する。So What? が自然に刺さらないなら、状況か複雑化のどちらかが弱い（この感覚は構造フェーズのステップ 5 と共通、詳細は `structure-topdown-8steps.md`）。

技術記事では、読者がすでに試したこと・引っかかったポイント・新しい要件などが複雑化の素材になる（企画フェーズの「読者がすでに試したこと」と対応）。

### 疑問 (Question)

疑問は **文書全体の中心軸** であり、文書内に **常にひとつしか存在しない**。この一つの疑問に答えるためにピラミッド全体が存在する。

書き下すか暗黙に留めるかは構成パターンによる。変形パターンのうち積極型では明示的に書き下し、他のパターンでは言外に隠れる場合もある。ただし、**書き手は書き始める前に必ず自分自身のために疑問を書き出し、絶対的確信を持つ** こと。書き出して確信を持てていない疑問のまま本論に入ると、書き手自身が「何に答える記事なのか」を自覚できず、下流が崩れる。

### 答え (Answer)

導入部の答えは **常に文書の主メッセージ** であり、ピラミッド頂上の主ポイントと **完全一致** する。書き手が文書を書く目的は、問題を解決するためか、すでに読み手の頭の中にある疑問に答えるためのどちらかで、いずれにせよ導入部で提示する答えと本論の主ポイントがズレることはない。技術記事では、R1 から R2 に到達するための道筋がこの答えにあたる。

ちなみに書籍は「答え」を **「解決」** と表記する箇所がある。変形パターンの呼称（解決 → 状況 → 複雑化など）はこの表記に合わせて使われる。

## 長さと粒度の原則

導入部の長さは本文の長さとは無関係。書き手と読み手が「同じ土俵に立っている」ことを確信できる長さがあれば十分で、**通常 2〜3 段落、まれに 3〜4 段落、それ以上にしてはいけない**。

技術記事における含意として、「はじめに」の肥大化は明確なアンチパターン。読者が知らない技術や概念を導入部で詳説し始めた時点で、それは「思い起こさせる」ではなく「知識を与える」になっており、導入部の役割から外れている。導入部が長くなりそうなときは、必要な前提知識が多すぎないか（想定読者と記事のスコープが合っているか）を企画に戻って見直すほうが正しい対処。

## 4 つの変形パターン

要素の並び順を変えた構成として、書籍は 4 つのパターンを独自の呼称で示す。「答え」は当該箇所では「解決」と表記される。

- **標準型**: 状況 → 複雑化 → 解決（S → C → 解決）
- **直接型**: 解決 → 状況 → 複雑化（解決 → S → C）
- **懸念型**: 複雑化 → 状況 → 解決（C → S → 解決）
- **積極型**: 疑問 → 状況 → 複雑化（Q → S → C。そのあとに解決が続く）

書籍はどのパターンを使うべきかの厳密な条件は設定しておらず、選択基準は **「その文書で伝えたいトーン（調子）」** と **「効果」**。配列順を変えると文書のトーンが変化するので、書き手が意図する印象に合わせて選ぶ。選択フローとしては、まず 4 要素（S/C/Q/解決）をすべて洗い出したうえで、**「効果に応じて好きな順序に配列してまったくかまわない」**。

技術記事での典型的な対応づけは以下のとおり。

- **標準型**: チュートリアル、概念解説記事。段階的な合意から導く王道
- **直接型**: TIL 系、「〜の方法」系。結論先行で読者の時間を節約
- **懸念型**: 「〜のアンチパターン」「〜をやめたほうがいい」系。危機感を先に打ち出す
- **積極型**: 読者の既知の疑問を掘り下げる探求系。問いから始めて読者に考えさせる

## 最重要の失敗警告: 考える順序まで変形した配列に合わせてしまう

変形パターンを使うときに書き手が陥りがちな最大の失敗は、**「考える順序」まで変形した配列に合わせてしまうこと**。実際の文書を直接型や懸念型で書く場合でも、**考える際には常に「状況」からスタートする** 原則を守らなければならない。最初から変形順で考えようとすると、正しい複雑化や疑問に到達できなくなる。

つまり、**思考の順序（常に S → C → Q → A）と、表現の順序（自由）は分離される**。対話フローでは常に標準型で思考を進め、パターン選択は最後の「表現化」段階で問う。この分離を守らないと、変形パターンが「書き手の混乱の隠れ蓑」になってしまう。

## 標準形でも起きる失敗警告

標準形で書く場合も、次の失敗が書籍で名指しされている。

- **導入部を省略してキーラインに飛ぶ**: 主ポイントから生じる疑問が明白に思えて、いきなりキーラインから書き始めてしまう失敗。結果として本来導入部に属すべき状況・複雑化の情報が本文に混ざり、展開が複雑になる
- **過去の出来事を本文に書く**: 単なる過去の出来事は「考え」ではないので、常に導入部に書くべき
- **「背景」や「はじめに」を本文の章として設ける**: これらの見出しは本文のキーライン・ポイントと同じレベルに属さない。下部キーラインと並列にしてはいけない
- **読み手と情報がズレる**: 読み手の知らないことを導入部に含めると疑問が歪む。逆に読み手がすでに知っている情報を本文 Q&A に含めると、導入部で書くべき情報を書き漏らした証になる

技術記事で特に多発するのが 3 番目の「背景」章の設置と、「はじめに」に前提知識を詰め込むパターン。いずれも導入部の役割（思い起こさせる）を逸脱しているので、検証フェーズでフラグを立てる対象。

## 対話フローとしての運用

導入フェーズのオーケストレーターは、思考と表現の分離を守るため **常に標準型（S → C → Q → A）で思考を進める**。

企画フェーズと構造フェーズの成果物がすでにあれば、ほとんどの要素は埋まっている。スタートポイント + 懸念される出来事 = S、読者がすでに試したこと = C、R1/R2 ギャップから導かれる問い = Q、R1 から R2 への道筋 = A、と割り当てた上で、So What? テストで S → C の接続を確認し、C → Q が即座に生じるかを再チェックする。

標準型で思考が固まったら、最後にパターン選択を問う。**「どのトーンで読者に提示したいか」** を書き手に選ばせ、選ばれたパターンに従って表現時の配列を決める。パターンを先に選ばせてはいけない（思考順序の汚染を招く）。

長さが 4 段落を超えそうなときは、「知識を与える」になっていないかを警告する。読者の前提知識と記事のスコープが合っていないサインなので、企画フェーズに戻る選択肢を提示する。
