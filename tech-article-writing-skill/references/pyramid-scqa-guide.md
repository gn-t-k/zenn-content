# ピラミッド原則とSCQAフレームワーク リファレンス

## ピラミッド原則（The Minto Pyramid Principle）

### 概要

バーバラ・ミントが開発した、論理的な文章構成のフレームワーク。**結論を先に述べ、その後に根拠を展開する**トップダウン構造を採用する。

### 基本構造

```text
           ┌─────────────┐
           │   結論      │  ← 最も重要なメッセージを最初に
           └─────────────┘
                 │
       ┌─────────┼─────────┐
       ▼         ▼         ▼
   ┌───────┐ ┌───────┐ ┌───────┐
   │キー   │ │キー   │ │キー   │  ← 結論を支える主要な論点
   │ポイント│ │ポイント│ │ポイント│
   └───────┘ └───────┘ └───────┘
       │         │         │
       ▼         ▼         ▼
   ┌───────┐ ┌───────┐ ┌───────┐
   │詳細   │ │詳細   │ │詳細   │  ← 各論点を支える詳細情報・根拠
   │情報   │ │情報   │ │情報   │
   └───────┘ └───────┘ └───────┘
```

### 3つの原則

| 原則 | 説明 |
| ----- | ------ |
| **結論から始める** | もっとも重要なメッセージを最初に伝える（BLUF: Bottom Line Up Front） |
| **グループ化** | 関連する情報をまとめ、キーポイントとして提示する |
| **論理的順序** | 情報を時系列、重要度、因果関係などで整理する |

### 技術記事での適用例

**悪い例（ボトムアップ）**:
> まずReactのuseStateを学び、次にuseEffectを学び、そしてカスタムフックを学びました。これらを組み合わせることで、複雑な状態管理が可能になります。

**良い例（トップダウン）**:
> Reactで複雑な状態管理を実現するにはカスタムフックが有効です。カスタムフックは、useState・useEffectなどの基本フックを組み合わせて再利用可能なロジックを作成する手法です。

## SCQAフレームワーク

### 基本概念

読者を問題認識から解決策へと自然に導くための導入部構成フレームワーク。ピラミッドの「入口」として機能する。

### 4つの要素

| 要素 | 英語 | 役割 | 例 |
| ----- | ------ | ------ | ----- |
| **S** | Situation | 読者が共感できる現状・背景を提示 | 「フロントエンド開発ではAPIからデータを取得する処理が頻繁に発生します」 |
| **C** | Complication | 状況における課題・問題を提起 | 「しかし、ローディング状態、エラーハンドリング、キャッシュ管理を毎回実装するのは煩雑です」 |
| **Q** | Question | 読者が持つべき問いを明示 | 「これらを簡潔に扱う方法はないでしょうか？」 |
| **A** | Answer | 解決策・結論を提示（＝記事の主題） | 「TanStack Queryを使えば、これらの課題をシンプルに解決できます」 |

### SCQAの流れ

```text
[S] 読者の共感を得る
      ↓
[C] 問題意識を喚起する
      ↓
[Q] 疑問を持たせる
      ↓
[A] 答え（記事の結論）を提示
      ↓
[本論] 結論の詳細をピラミッド構造で展開
```

### SCQA適用例

#### 例1: ライブラリ紹介記事

```markdown
## はじめに

フロントエンド開発において、フォームのバリデーションは避けて通れない課題です。**[S]**

しかし、バリデーションルールの定義、エラーメッセージの管理、再レンダリングの最適化など、
自前で実装しようとすると複雑になりがちです。**[C]**

もっとシンプルにフォームバリデーションを実装できないでしょうか？**[Q]**

本記事では、React Hook Formとzodを組み合わせることで、
型安全かつ宣言的なフォームバリデーションを実現する方法を紹介します。**[A]**
```

#### 例2: トラブルシューティング記事

```markdown
## はじめに

Next.jsのApp Routerを使用していると、Server ComponentsとClient Componentsの
使い分けが必要になります。**[S]**

先日、`useState`を使ったコンポーネントで「This component should be a Client Component」
というエラーに遭遇しました。単に`'use client'`を追加すれば解決すると思いきや、
予期せぬ副作用が発生しました。**[C]**

Server/Client Componentsの境界を正しく設計するにはどうすればよいのでしょうか？**[Q]**

本記事では、エラーの根本原因と、パフォーマンスを損なわない正しい対処法を解説します。**[A]**
```

## ピラミッド原則とSCQAの関係

```text
┌─────────────────────────────────────────────────┐
│                    記事構造                      │
├─────────────────────────────────────────────────┤
│  【導入部】← SCQAフレームワーク                  │
│    S: 状況設定（読者の共感を得る）               │
│    C: 問題提起（課題を認識させる）               │
│    Q: 疑問の喚起（問いを持たせる）               │
│    A: 答え ─────────────────────┐               │
├─────────────────────────────────│───────────────┤
│  【本論】← ピラミッド原則       │               │
│                                 ▼               │
│              ┌─────────────────────┐            │
│              │  結論（= Answer）   │            │
│              └─────────────────────┘            │
│                       │                         │
│         ┌─────────────┼─────────────┐           │
│         ▼             ▼             ▼           │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐      │
│    │ キー    │   │ キー    │   │ キー    │      │
│    │ポイント1│   │ポイント2│   │ポイント3│      │
│    └─────────┘   └─────────┘   └─────────┘      │
│         │             │             │           │
│         ▼             ▼             ▼           │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐      │
│    │ 詳細    │   │ 詳細    │   │ 詳細    │      │
│    └─────────┘   └─────────┘   └─────────┘      │
├─────────────────────────────────────────────────┤
│  【まとめ】                                      │
│    結論の再確認、次のアクション                  │
└─────────────────────────────────────────────────┘
```

**SCQAは「なぜこの記事を読むべきか」を伝え、ピラミッド原則は「何をどう伝えるか」を構成する。**

## チェックリスト

### SCQA導入部のチェック

- [ ] Situationは読者が「わかる」と思える内容か
- [ ] Complicationは具体的で、読者が「困る」と感じるか
- [ ] Questionは読者の頭に自然に浮かぶ問いか
- [ ] AnswerはQuestionへの直接的な回答になっているか

### ピラミッド構造のチェック

- [ ] 結論は1文で表現できるか
- [ ] キーポイントは結論を支える根拠になっているか
- [ ] キーポイント同士に重複がないか（MECE）
- [ ] 詳細情報は各キーポイントを裏付けているか
- [ ] 情報の粒度は揃っているか
