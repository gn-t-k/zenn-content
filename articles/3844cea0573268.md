テーブル設計で「この属性は状況によって null になる」と素直に nullable にしてしまうことは、よくあります。特に「種類ごとに必須属性が変わる」「N 個のうち K 個を指定する」といった組み合わせ制約が絡むと、nullable なカラムがどんどん増えていきます。

でも nullable の多いテーブルは、**DB が不正な組み合わせを黙って受け入れる**設計になりがちです。アプリのバリデーションで書き漏れや条件ミスがあれば、沈黙のうちに壊れたデータが溜まっていきます。

この記事では、**そもそも不正な状態を作れないように、テーブル構造そのもので組み合わせを強制する**設計を、筋トレアプリで遭遇した実例から紹介します。関数型プログラミングで `Make illegal states unrepresentable`（不正な状態を表現不可能にする）という言葉で知られる発想の、DB 設計版です。ただし構造だけで防ぎきれない部分もあるので、その代償もあわせて書きます。

## 筋トレアプリで遭遇した「3つのうち2つ」問題

トレーニング計画を管理するアプリを個人開発していて、こんな要件にぶつかりました。

> 1セット分のパラメータは、**重量・レップ数・RPE** の3つのうち**2つを指定**する。

:::message
RPE は Rate of Perceived Exertion の略で、0〜10の主観的運動強度のことです。「あと何レップ余裕があるか」を表す指標で、RPE 8 なら「あと2レップできそう」くらいの負荷感を指します。
:::

組み合わせは3パターンあります。

- **重量 + レップ数**: 「60kg を 5回」のように、挙げる重量と回数を指定する。一番オーソドックス
- **重量 + RPE**: 「60kg を RPE 8 まで」のように、重量を決めて「そのくらいきつくなるまで」やる
- **レップ数 + RPE**: 「RPE 8 で 5回」のように、回数を決めて、それが RPE 8 になる重量を選ぶ

どれを2つ選ぶかで、そのセットの実行方法が変わります。

素直にテーブルを作ろうとすると、こうなります。

```sql
CREATE TABLE set_plans (
  id               TEXT PRIMARY KEY,
  exercise_plan_id TEXT NOT NULL,
  weight           REAL,     -- nullable
  reps             INTEGER,  -- nullable
  rpe              REAL      -- nullable
);
```

`weight / reps / rpe` のどれが入るかはレコードごとに違うので、**3つとも nullable** にするしかなさそうに見えます。

読み出しクエリは素直で、結果の型はこうなります。

```sql
SELECT id, weight, reps, rpe FROM set_plans WHERE id = ?;
```

```ts
type SetPlan = {
  id: string;
  weight: number | null;
  reps: number | null;
  rpe: number | null;
};
```

3カラムとも nullable なので、読み出し側のコードも「null かもしれない」という前提で書くことになります。ところが、この設計にはすぐに破綻の兆しが現れます。

### DB はどんなレコードでも受け入れてしまう

3カラムが全部 nullable ということは、DB は次のようなレコードも**すべて受け入れる**ということです。

| weight | reps | rpe | 状態 |
|---|---|---|---|
| 60 | 5 | null | 正常（重量+レップ） |
| null | 5 | 8 | 正常（レップ+RPE） |
| null | null | null | **3つとも空** — 計画として成立していない |
| 60 | 5 | 8 | **3つとも入力** — どう実行する想定？ |
| 60 | null | null | **1つしか入力されていない** — 何をすればいいか決まらない |

下3行のレコードが混入すると、アプリは「このセットをどう表示するか」「記録画面にどう誘導するか」を決められません。表示がバグる、クラッシュする、黙ってスキップする、といった症状が出ます。

### 問題は「気づけないこと」

バリデーションに書き漏れや条件ミスがあったとき、DB が不正な組み合わせを受け入れる設計だと、**DB はエラーを返しません**。不正なデータが成功レスポンスを返しながら書き込まれ続け、気づくのは数週間後、「最近セット計画の挙動が変」と感じた頃です。混入した不正データはテストでは見つからず、DB を直接のぞき込んで修復する羽目になります。

しかも DB を信用できなくなれば、**読み出し側にも防御コードが染み出します**。「このレコードの組み合わせは本当に正しいか」を毎回確認するコードが、読み書きの両側で増えていきます。

そして個人開発でも、API を通らない経路は遅かれ早かれ増えます。管理画面、データ移行バッチ、第三者ツールからの直接アクセス。そのたびに同じバリデーションを同じ正確さで守り続ける責任が、アプリ側に降ってきます。

**気づけないバグは、あとで直すのが一番高くつきます**。

## サブタイプテーブル分割で「ありえない状態」を構造から追放する

冒頭で触れた `Make illegal states unrepresentable` を DB スキーマに持ち込んだのが、サブタイプテーブル分割です。具体的には、`set_plans` を**基底テーブル**と**3つの子テーブル**に分割します。

```sql
-- 基底テーブル: どの組み合わせなのかだけを持つ
CREATE TABLE set_plans (
  id               TEXT PRIMARY KEY,
  exercise_plan_id TEXT NOT NULL,
  plan_type        TEXT NOT NULL  -- 'weight_reps' / 'weight_rpe' / 'reps_rpe'
);

-- 重量 + レップ数
CREATE TABLE weight_reps_params (
  set_plan_id TEXT    PRIMARY KEY REFERENCES set_plans(id),
  weight      REAL    NOT NULL,
  reps        INTEGER NOT NULL
);

-- 重量 + RPE
CREATE TABLE weight_rpe_params (
  set_plan_id TEXT PRIMARY KEY REFERENCES set_plans(id),
  weight      REAL NOT NULL,
  rpe         REAL NOT NULL
);

-- レップ数 + RPE
CREATE TABLE reps_rpe_params (
  set_plan_id TEXT    PRIMARY KEY REFERENCES set_plans(id),
  reps        INTEGER NOT NULL,
  rpe         REAL    NOT NULL
);
```

基底テーブル `set_plans` は、weight/reps/rpe の値そのものは持たず、**どの組み合わせなのか**を `plan_type` カラムに記録するだけの役割に徹します。`plan_type` には `'weight_reps' / 'weight_rpe' / 'reps_rpe'` のいずれかが入ります。値は3つの子テーブルのうち、`plan_type` が指し示すテーブルに格納します。子テーブルはその組み合わせに必要な**2カラムだけを持ち、両方とも NOT NULL** です。

読み出しは、まず `set_plans` から `plan_type` を取得し、その値に応じて対応する子テーブルに SELECT を投げます。

```sql
-- 1回目: plan_type を取得
SELECT id, plan_type FROM set_plans WHERE id = ?;

-- 2回目: plan_type = 'weight_reps' ならこのクエリ
SELECT weight, reps FROM weight_reps_params WHERE set_plan_id = ?;
```

アプリ側は `plan_type` で分岐させれば、各分岐のなかで取得したカラムはそのまま NOT NULL として扱えます。戻り値は `plan_type` ごとに形が変わる1つの型として表現できます。

```ts
type SetPlan =
  | { id: string; planType: 'weight_reps'; weight: number; reps: number }
  | { id: string; planType: 'weight_rpe';  weight: number; rpe: number  }
  | { id: string; planType: 'reps_rpe';    reps: number;   rpe: number  };
```

`planType` で分岐すれば、その中で使えるプロパティが決まるので、null を意識した書き方は要りません。DBから取れた値をそのまま型に当てはめられます。

### 不正な組み合わせが「そもそも作れない」

Before で不正だった「3つとも入力」や「1つしか入力されていない」といったパターンは、After では**レコード単位で構築自体が不可能**です（テーブルをまたいだ整合性の話は後述のトレードオフで扱います）。

- **「3つとも入力」**: 1つの子テーブルには該当2カラムしか存在しないので、3つ目の値を書き込む箱がありません。SQL として書けません
- **「1つしか入力」**: どの子テーブルを選んでも、もう1つの NOT NULL カラムを埋めないかぎり INSERT が失敗します

アプリ側でバリデーションロジックを書いて防いでいるのではなく、**テーブル構造がその入力を物理的に受け付けません**。これが違いです。バリデーションに書き漏れや条件ミスがあっても、DB が INSERT を拒否してくれます。**バリデーションコードが減る = バグの入り込む余地そのものが減る**ので、Before で怖がっていた「沈黙するバグ」は、そもそも成立しません。

さらに、この設計は Before が区別できなかった2つの状態を自然に分離してくれます。`plan_type` は入っているが子テーブルにまだ行がない状態は、**組み合わせパターンは決めたが、具体値はまだ入力していない途中保存**として扱えます。Before では「不正レコード」とも「途中保存」ともつかなかった「3つとも空」の状態が、After では意味のある状態として表現できるようになった、とも言えます。

## トレードオフ

サブタイプテーブル分割は万能ではありません。代償もいくつかあります。

### テーブル数が1から4に増える

単純な話ですが、テーブル1つで済んでいたものが4つになります。マイグレーションファイル、ER図、ドキュメントはその分だけ膨らみます。

### 読み出しでラウンドトリップが増える

`id` だけで1件読み出す場合、Before は1回の SELECT で完結していましたが、After は**2回の SELECT が必要**です。

1. `set_plans` から `plan_type` を取得
2. `plan_type` に応じた子テーブルから値を取得

3つの子テーブルを LEFT JOIN でまとめて1回で取る書き方もあります。ただしその場合、結果は nullable なカラムが並ぶ平たい行になるので、型として綺麗に扱うにはアプリ側で `plan_type` を見て詰め直す変換コードが必要です。その変換のなかでは「この値は null ではないはず」を強引に主張するような書き方が入り込みやすく、型の安全性は2回 SELECT する場合より弱くなります。**型の安全さを優先するなら 2 回 SELECT**、**ラウンドトリップ1回を優先するなら LEFT JOIN + 変換コード**、というトレードオフです。

### 「ちょうど1つの子テーブルに行が存在する」は DB では保証できない

正直に言っておくべき制約があります。`set_plans.plan_type = 'weight_reps'` のとき、**対応する `weight_reps_params` にだけ行があり、他の子テーブルには行がない**、という排他的な状態は、DB 構造だけでは強制できません。たとえば次のような**幽霊データ**（plan_type が指さない子テーブルに残ってしまった行）を、作ろうと思えば作れてしまいます。

- `plan_type = 'weight_reps'` だが、実際には `weight_rpe_params` に行がある
- 同じ `set_plan_id` で、`weight_reps_params` と `reps_rpe_params` の両方に行がある

ただし、幽霊データはアプリからは基本的に見えません。アプリは `plan_type` が指す子テーブルしか JOIN しにいかないので、**対応しない子テーブルに残ったデータは読み出されず、DB 内に静かに居座るだけ**です。ユーザー画面にはとくに症状が出ません。

問題として現れるのは、データ側を直接触った場面です。バックアップやデータ移行で全テーブルを舐めたとき、あるいは `plan_type` を別の値に変更した結果、以前に書かれた別テーブルの行が急に「見える」ようになったとき。このタイミングで「なぜこんなデータがあるんだ」と気づきます。

この排他性を守るには、アプリ側の書き込み経路で「**`plan_type` に対応する子テーブルにだけ行を入れる**」というルールを貫く必要があります。書き込み経路を1箇所にまとめていれば、それほど難しくはありません。Before のように nullable 3カラムを全方位で管理するのと比べれば、守るべきルールは明確で狭いとも言えます。

## 自分のスキーマに持ち帰る

この記事のエッセンスをひと言でまとめると、こうなります。

**nullable カラムが多いテーブルを見たら、その裏に組み合わせ制約が隠れていないかを疑う**。

筋トレの「2 of 3」は具体例の1つにすぎません。似たパターンは他のドメインでも姿を変えて現れます。

- **商品タイプで必須属性が変わる**: `digital` ならダウンロード URL 必須、`physical` なら配送先住所必須
- **支払い手段で必須情報が違う**: クレジットカードなら番号、銀行振込なら口座情報
- **通知チャネルが選択制**: メールなら宛先アドレス、Webhook なら送信先 URL

いずれも「区分カラム + 区分ごとの固有属性」という同じ構造で、素直に1テーブルに収めると nullable だらけになります。サブタイプ分割すれば、**組み合わせを構造そのものが強制する**設計に持っていけます。

`Make illegal states unrepresentable` は関数型プログラミングの型システムの文脈で語られる考え方でしたが、発想自体は DB 設計にも移植できます。nullable が増えそうだと感じたら、バリデーションコードを書く前に「このルール、スキーマ側で表現できないか？」を一度考えてみる。それだけで、沈黙するバグが忍び込む余地は目に見えて狭まります。
