個人開発で筋トレアプリを開発しています。事前に筋トレの計画を立てる機能と、トレーニング時に記録を保存する機能があります。

1セット分の計画を立てる場合、3パターンのいずれかの形式を選んで計画データを保存します。

|パターン|計画例|
|-|-|
|重量と回数を指定する|100kgを8回やる|
|重量とRPEを指定する|100kgでRPE8までやる|
|回数とRPEを指定する|RPE8になる重量を8回やる|

このような計画をもとにしてトレーニングを実施し、「100kgで8回できて、RPE7だった」のような記録を保存します。

:::details RPEとは
RPE (Rating of Perceived Exertion) は自覚的運動強度を表す 10 段階の指標のことです。「100kg で RPE8」という計画は、「100kg で『あと 2 回はできそう』な限界まで回数をこなす」という意図を表します。
:::

セット計画データをRDBに保存する場合、以下のようなテーブル設計が考えられます。

```sql
CREATE TABLE set_plans (
  id     UUID PRIMARY KEY,
  weight NUMERIC(6, 2),  -- nullable
  reps   INTEGER,        -- nullable
  rpe    SMALLINT        -- nullable
);
```

このテーブル設計の場合、`set_plans`テーブルからレコードを取得するクエリは次のとおりです。

```sql
SELECT id, weight, reps, rpe FROM set_plans WHERE id = $1;
```

そしてORM が返してくる行の型は次のようになります。

```typescript
type SetPlanRow = {
  id: string;
  weight: number | null;
  reps: number | null;
  rpe: number | null;
};
```

アプリケーションで`SetPlanRow`を計画として表示するために、このレコードが3パターンの計画のうちどのパターンに当てはまるのかを判定します。

```typescript
type SetPlan =
  | { id: string; kind: "weight_reps"; weight: number; reps: number }
  | { id: string; kind: "weight_rpe"; weight: number; rpe: number }
  | { id: string; kind: "reps_rpe"; reps: number; rpe: number }

const validateSetPlanRow = ({ id, weight, reps, rpe }: SetPlanRow): SetPlan => {
  if (weight !== null && reps !== null && rpe === null) {
    return { id, kind: "weight_reps", weight, reps };
  }
  if (weight !== null && reps === null && rpe !== null) {
    return { id, kind: "weight_rpe", weight, rpe };
  }
  if (weight === null && reps !== null && rpe !== null) {
    return { id, kind: "reps_rpe", reps, rpe };
  }
  throw new Error('計画のパターンが不正です');
};

```

このテーブル設計・アプリケーション実装には様々な問題が存在します。

この記事では、このテーブル設計・アプリケーション実装の弱点とその克服方法、トレードオフについて紹介します。

## nullable + バリデーション設計の弱点

はじめに挙げた問題はDDLにCHECK制約を書いていないことに起因します。ではCHECK制約を追加すれば問題は解決するのでしょうか。CHECK制約を書くかどうかは、組み合わせ制約の保証責任をDBとアプリのどちらに持たせるかの選択であり、どちらを選んでも形の違うコストが残ります。

### 読み込みパスごとにバリデーションが必要

まず、アプリケーション実装のバグがあった場合、DBに不正データが保存される可能性があります。そうなった場合、アプリケーションで不正データを読み込んでしまったときに実行時エラーになる。**実行時エラーを防ぐため、すべての読み込み箇所でデータのバリデーションが必要になります。**

### 書き込みパスごとにバリデーションが必要

次に、この設計・実装の場合、組み合わせの正しさを保つ責任はすべてアプリ側に移ります。Web APIのCreateエンドポイント、バッチワーカー、管理画面からの登録、データ移行スクリプトなど、どれもDBにとっては等しく「書き込み要求」で、どの経路から書かれても同じ「3パターンのいずれか」検査を通す必要があります。

すべての書き込み要求のペイロードを検査するのは、運用上は正しい要求です。ただし、**同じ概念を複数の経路で書かなければいけないこと自体が問題**になります。規律を守りそこねた経路が1つあれば、そこから不正な組み合わせがDBに入ります。

### CHECK 制約を書く場合: バリデーションは一元化できるが、プロダクションコードとテストの凝集度が下がる

DB側で組み合わせを弾けば、書き込み経路が複数あってもDBレベルで守られます。CHECK制約は各書き込みパスのバリデーション漏れに対する「最後の関所」として働きます。

```sql
CREATE TABLE set_plans (
  id     UUID PRIMARY KEY,
  weight NUMERIC(6, 2),
  reps   INTEGER,
  rpe    SMALLINT,

  CHECK (
    (weight IS NOT NULL AND reps IS NOT NULL AND rpe IS NULL)     OR
    (weight IS NOT NULL AND reps IS NULL     AND rpe IS NOT NULL) OR
    (weight IS NULL     AND reps IS NOT NULL AND rpe IS NOT NULL)
  )
);
```

代わりに別の問題が出ます。アプリケーションコードとテストコードは同じパッケージに置かれ、PRレビューでもセットで更新されます — プロダクションの挙動を変える変更には、対応するテストの変更が自然と並びます。対してCHECK制約はマイグレーションファイルにあり、挙動を保証するには別途統合テスト環境が必要で、書き場所が物理的に離れています。制約の仕様（プロダクションコード）とその挙動を保証するもの（テスト）の凝集度が下がります。結果として「CHECKは書いたがテストはない」「CHECK式を変えたがテストが追随していない」という状態が残りやすくなります。

## サブタイプテーブル分割で不正な組み合わせを構造的に排除する

ナイーブ設計の各問題に共通していたのは「有効な組み合わせは3通りしかないのに、スキーマは全パターンを表現できてしまう」点でした。**組み合わせごとに別テーブルを作り、各テーブルは全カラムNOT NULLにすれば**、スキーマが表現できる形そのものを有効な組み合わせだけに絞れます。

```sql
CREATE TYPE plan_type AS ENUM ('weight_reps', 'weight_rpe', 'reps_rpe');

-- 基底テーブル: どのサブタイプかを保持
CREATE TABLE set_plans (
  id        UUID PRIMARY KEY,
  plan_type plan_type NOT NULL
);

-- サブタイプ 1: weight + reps
CREATE TABLE weight_reps_params (
  set_plan_id UUID PRIMARY KEY REFERENCES set_plans(id) ON DELETE CASCADE,
  weight      NUMERIC(6, 2) NOT NULL,
  reps        INTEGER       NOT NULL
);

-- サブタイプ 2: weight + rpe
CREATE TABLE weight_rpe_params (
  set_plan_id UUID PRIMARY KEY REFERENCES set_plans(id) ON DELETE CASCADE,
  weight      NUMERIC(6, 2) NOT NULL,
  rpe         SMALLINT      NOT NULL
);

-- サブタイプ 3: reps + rpe
CREATE TABLE reps_rpe_params (
  set_plan_id UUID PRIMARY KEY REFERENCES set_plans(id) ON DELETE CASCADE,
  reps        INTEGER  NOT NULL,
  rpe         SMALLINT NOT NULL
);
```

構造のポイントは3つです。

- 基底テーブル `set_plans` は `plan_type` を持つ。この列が「このレコードはどのサブタイプか」を示す **discriminator** として働く
- サブタイプテーブルの主キーは `set_plan_id`。基底と 1:1 対応し、独立した id は持たない
- 各サブタイプテーブルは**全カラム NOT NULL**。その組み合わせに属する列だけが存在し、欠けるはずの列はそもそもスキーマに存在しない

この構造の上では、前章で挙げたナイーブ設計の問題がそれぞれ消えます。

- **書き込みパスごとのバリデーション**: 不要。「3 つとも値あり」「片方だけ入力」のような不正な組み合わせは単一テーブルに存在しないので、経路を問わず書けない
- **プロダクションコードとテストの凝集度**: 問題にならない。CHECK 式そのものが要らないので、マイグレーションに切り離された制約仕様を抱えずに済む

ただし新しい対価も出ます。取得クエリはSELECT一発では済まなくなり、さらにDB構造だけでは消せない整合性問題が基底とサブタイプの間に残ります。

## サブタイプ分割のトレードオフ

サブタイプ分割は銀の弾丸ではありません。対価は主に2種類です: **取得クエリの複雑化** と、**基底とサブタイプの整合性管理**。

### 取得クエリの実装は 2 段階クエリか LEFT JOIN の二択

ナイーブ設計では `SELECT` 一発で済んだ取得が、サブタイプ分割では2段階クエリとLEFT JOINの二択になります。

#### 案 A: 2 段階クエリ

まず `plan_type` を取って、対応するサブタイプテーブルを取りに行きます。

```sql
-- 1 発目: plan_type を取得
SELECT plan_type FROM set_plans WHERE id = $1;

-- 2 発目: plan_type = 'weight_reps' のとき
SELECT weight, reps FROM weight_reps_params WHERE set_plan_id = $1;
```

各SQLがそれ自体で単純で、追いやすいです。さらに、**アプリ側の型も自然とnullableなしのdiscriminated unionになります**。

```typescript
type SetPlanParams =
  | { kind: "weight_reps"; weight: number; reps: number }
  | { kind: "weight_rpe";  weight: number; rpe: number }
  | { kind: "reps_rpe";    reps: number;   rpe: number };
```

2発目のSELECTが返す行はすでにNOT NULLのカラムしか含んでいないので、narrowingもアサーションも要りません。これは案Aの強い利点です。

対価はDBへのラウンドトリップが2回になることです。1件取得が中心のパスでは気になりませんが、N件の一覧取得で単純に繰り返すと効いてきます（ただし「plan_typeごとにまとめて一覧を取る」のような工夫で実用上は回避できます）。

#### 案 B: 全サブタイプに LEFT JOIN

1 SQLで取り切ります。列名衝突を避けるためエイリアスを付けます（`wr_*`がweight_reps_params、`we_*`がweight_rpe_params、`rr_*`がreps_rpe_params）。

```sql
SELECT
  sp.id, sp.plan_type,
  wr.weight AS wr_weight, wr.reps  AS wr_reps,
  we.weight AS we_weight, we.rpe   AS we_rpe,
  rr.reps   AS rr_reps,   rr.rpe   AS rr_rpe
FROM set_plans sp
LEFT JOIN weight_reps_params wr ON sp.id = wr.set_plan_id
LEFT JOIN weight_rpe_params  we ON sp.id = we.set_plan_id
LEFT JOIN reps_rpe_params    rr ON sp.id = rr.set_plan_id
WHERE sp.id = $1;
```

ラウンドトリップは1回です。`plan_type` の値に応じて3つのJOINのうち1つだけが非nullになることがDB構造で保証されています。対価は、結果が各列nullableで返ってくるのでORM層のマッピング関数に `!` が出てくることです。

```typescript
const mapRow = (row: RawRow): SetPlanParams => {
  switch (row.plan_type) {
    case "weight_reps":
      return { kind: "weight_reps", weight: row.wr_weight!, reps: row.wr_reps! };
    case "weight_rpe":
      return { kind: "weight_rpe",  weight: row.we_weight!, rpe:  row.we_rpe! };
    case "reps_rpe":
      return { kind: "reps_rpe",    reps:   row.rr_reps!,   rpe:  row.rr_rpe! };
  }
};
```

この `!` は、`plan_type === "weight_reps"` のとき `weight_reps_params` の行が必ず存在し、その全カラムがNOT NULLであることをDDLが担保しています — TypeScriptがそれを知らないだけです。

### JOIN 増によるパフォーマンス・保守性の低下は実質的に軽い

サブタイプ分割を提案すると、「テーブル数とJOINが増えて、パフォーマンスや保守性が下がる」という懸念が返ってくることが多いです。踏み込むと弱い懸念です。

パフォーマンス面では、`set_plan_id` がPKなので対応するindexは自動で張られます。JOINのコストは実質的に問題になりません。

保守性の面では、SQLが長くなること自体はそのとおりです。ただし各LEFT JOINが同じ形で並び、エイリアスが揃っているので、長さの割に構造は読み取りやすいです。テーブルが増えることによる変更時の同時更新コスト（DDL / SQL / ORMマッピングを一緒に触る）は素直に残りますが、これはサブタイプ分割そのものの弱点というより、テーブル数を増やす設計判断一般の対価として捉えるのが素直です。

### 基底とサブタイプの整合性はアプリが管理する

DB構造で完全に消せない不整合が1つ残ります。**基底の `plan_type` と、実際にレコードが存在するサブタイプテーブルが食い違う** パターンです。たとえば `plan_type = 'weight_rpe'` の基底レコードに対して `weight_reps_params` にレコードが入っている、という状態です。DB構造だけではこれを禁止できません。

ここはアプリ側が責任を持ちます。`set_plans` へのINSERTと該当サブタイプテーブルへのINSERTを同じトランザクションで束ね、DELETEも同じトランザクションで束ねます。トランザクションを使うこと自体はナイーブ設計でも普通にやる話なので負担の追加ではありませんが、「どのサブタイプテーブルに書くべきか」をアプリが `plan_type` と対応させて判断する責任は追加で発生します。

## nullable 列を見たときに問うてよいこと

この記事で扱ったのは「3つのうち2つ」という具体要件でしたが、軸は一段抽象化できます。サブタイプ分割を適用してよいかは、次の2軸で判断できます。

1. **組み合わせの種類が有限で列挙可能か**: 「3 つのうち 2 つ」なら 3 通りで有限。「100 列のうち任意の 2 つ」のように組み合わせ爆発するケースでは非現実的
2. **各組み合わせに NOT NULL にしたい列があるか**: 「この組み合わせならこの列は必ず埋まる」と言える列がある要件でこそ意味がある。「どの組み合わせでも全列 nullable で OK」なら、そもそも制約がないので分割する理由がない

両方当てはまるときが、サブタイプ分割の適用領域になります。nullable列が複数並んだテーブルを見たときに、「このnullableは本当に不定なのか、それとも有限個の組み合わせパターンに分解できるのか」を問うてみてください。後者なら、サブタイプ分割でスキーマから不正な組み合わせを排除できる可能性があります。

同じ2軸で適用可否を判定できる要件は他にも多くあります。たとえば「AかBのどちらか必須」「種別ごとに必須属性が違う」「排他的な状態遷移（ドラフト/公開/アーカイブで持つべき列が異なる）」など。

アプリ側バリデーションで頑張るか、DB構造そのものに制約を表現するか。型設計の原則として知られる "Make illegal states unrepresentable" は、DBスキーマ設計にもそのまま効きます。
